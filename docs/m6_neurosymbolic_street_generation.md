# M6: Neuralsymbolic Street Generation v1

## Overview

M6 refactors street composition from a direct `query -> slot placement` flow into:

```text
query/context
  -> StreetProgram
  -> ConstraintSet
  -> LayoutSolver
  -> asset realization
  -> mesh export
```

The goal is to make street structure and design rules explicit, editable, and testable before 3D realization.

## New Public Interfaces

- `StreetProgram`
  - structured street intent including road type, cross section, functional bands, furniture requirements, control points, and design goals
- `DesignRuleSpec` / `ConstraintSet`
  - declarative street-design rules with hard/soft mode and explainable targets
- `LayoutSolverInput` / `LayoutSolverResult`
  - constraint-compilation boundary between street intent and placement realization

## Current Built-In Rule Profiles

- `balanced_complete_street_v1`
- `pedestrian_priority_v1`
- `transit_priority_v1`

Each profile can:
- widen or narrow cross-section bands
- change category demand targets
- reserve or ban categories in specific band kinds
- emit explicit `move`, `remove`, or `replace` edits

## Output Additions

`scene_layout.json` now includes:
- `street_program`
- `constraint_set`
- `solver`

Summary adds:
- `rule_satisfaction_rate`
- `topology_validity`
- `cross_section_feasibility`
- `editability`
- `conflict_explainability`
- `solver_edit_count`
- `solver_conflict_count`

## Current Limits

- `StreetProgram` is still generated by `heuristic_v1`; it is not yet trained end-to-end.
- `LayoutSolver` is a banded heuristic solver, not a MILP or diffusion-based layout model.
- OSM mode still uses the existing geometry/POI path; the new program and rule layers mainly structure the template and realization flow.
