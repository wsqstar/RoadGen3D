STYLE DIAGNOSIS

RoadGen3D 已经像工程系统，下一步是让控制台像工作台。

当前界面有完整的生成、评估、对比、历史、资产和 Viewer 能力，但“流程页”和“场景画布”分散。优化重点不是删功能，而是把功能按任务上下文重排：左侧浏览对象，中间控制生成，右侧分析结果，底部追踪任务与产物。

建议方向
  • 视觉系统 保留道路/城市主题蓝和评估语义色，但降低大面积蓝色和卡片阴影。
  • 信息架构 从三步线性页面转为 Viewer 工作台，分析层随场景切换同步。
  • 交互策略 展开/收起不是省空间装饰，而是把低频分析和高级设置变成上下文抽屉。

现有风格判断

旧 Workbench 使用明亮 HKUST 蓝、Ant Design 组件、三步流程和卡片网格；当前 Viewer 已经出现更接近桌面工作台的左右栏、设置面板、Design/Evaluate/Compare/History 右侧标签。整体方向应从“演示型卡片应用”转向“专业场景生成控制台”。

颜色

当前蓝黄主题识别度高，但蓝色同时承担品牌、按钮、状态、选中，导致层级偏吵。

HKUST 蓝保留为域色 绿色负责成功/就绪 黄色只做缺失/等待

布局

旧 Workbench 的 1200px 居中容器适合说明页，不适合长期运行和并行分析。

  • 保留三步流程、方案卡、评估卡、对比表。
  • 调整把流程拆成栏位和抽屉，减少页面跳转。

密度

开发者需要同时看到布局、任务、指标和产物路径；界面应提高信息密度。

单屏保留上下文 指标使用等宽数字 高级项默认折叠

功能面不能删，只能重排

原能力 现状入口 建议归位 原因
Recent Layouts / Scene Viewer 左侧入口 左侧浏览栏常驻 开发者切场景时不能丢失右侧评估和任务上下文。
Design Assistant Viewer 右侧 Design 面板 中间控制面板 + 可折叠高级项 结构、家具、矩阵、补充要求是生成控制，不应被藏在纯分析侧栏。
Evaluate / Compare / History Viewer 右侧标签 右侧分析栏 评估、对比、历史都依赖当前场景，应随画布同步更新。
Scene Job operations 状态文案和生成过程按钮 底部任务轨迹 长任务需要可展开事件流，显示 stage、progress、error、result 路径。
Round-trip / bridge report CLI 与 JSON 报告语义 分析栏的“数据一致性”模块 保留 status、geo_delta、node_recall、edge_recall、topology_ok、loss_digest 语义。

展开/收起的正确用法

展开/收起应该反映任务频率，而不是简单隐藏内容。RoadGen3D 的控制面板可以用四层来组织：常驻、悬停展开、按需抽屉、任务轨迹。

层级 放什么 默认状态 设计理由
常驻层 当前 Layout、Scene、生成按钮、画布、核心评分 展开 这些信息决定当前任务，不应被用户找来找去。
浏览栏 最近布局、场景树、对象层、资产入口 桌面展开，窄屏收起 浏览对象需要连续操作，但移动端必须让画布优先。
分析栏 评估、对比、历史、Benchmark、LLM 状态 桌面展开，可固定；中屏抽屉 分析是高价值次级任务，应该随场景同步但不遮挡画布。
底部轨迹 任务事件、stage、progress、产物路径、错误 收起为状态条 长任务和失败排查需要完整轨迹，但不应常年占据主视野。

落地建议

以 Viewer 为壳

不要恢复独立 Workbench 作为主体验。

  • 保留 Viewer3D 场景、Scene Graph、Annotation、Asset Editor 继续作为主画布生态。
  • 迁移语义旧 Workbench 的三步生成、方案卡、评估卡可以变成 Viewer 面板内容。

让状态可解释

开发者最需要知道任务为什么成功、失败或等待。

  • 任务轨迹展示 created、started、stage、progress、finished、error 和 result。
  • 一致性指标桥接检查保留 conversion_ok、geo_delta、topology_ok、loss_digest。
